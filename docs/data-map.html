02

연결 구조 — 세 개의 좌표계

가장 중요한 사실: 이 10개 소스는 하나의 통합 ID 체계를 쓰지 않습니다. 서로 다른 두 “코드 우주”가 있고, 그 사이는 이름 매칭으로만 이어지며, 지역 단위에서는 별도의 공통 배관(법정동 코드)이 모두를 묶습니다.

실선 = 코드 직접 조인 · 갈색 점선 = 이름 매칭(불완전) · 회색 점선 = 코드 조합식 필요

코드 직접 조인 (신뢰도 높음) 이름 텍스트 매칭 (불완전, fuzzy) 조합식 필요 (formula join)

04

조인 전략 — 실제 쿼리 예시

신뢰도가 높은 것부터 낮은 것 순서로 정리했습니다. 위쪽 두 개는 그대로 써도 되고, 아래쪽은 결과를 검증하며 쓰세요.

① 연관관광지 ↔ 기초지자체중심 (코드 직접 조인)

신뢰도: 높음 · 74.5%
select r.t_ats_nm, r.rlte_tats_nm, h.hub_rank
from tar_rlte_records r
join locgo_hub_records h
  on h.hub_tats_cd in (r.t_ats_cd, r.rlte_tats_cd)
 and h.base_ym = r.base_ym;

두 서비스가 같은 관광지 해시코드 마스터를 공유하는 것을 확인했습니다(전체 기준 74.5%). 이름이 아니라 코드로 묶이므로 동명이인/표기차 문제가 없습니다.

② kor_poi 지역코드 정규화 (법정동 5자리 만들기)

신뢰도: 높음 · 검증됨
select
  content_id, title,
  ldong_regn_cd || lpad(ldong_signgu_cd, 3, '0') as signgu_cd_5
from kor_poi
where ldong_signgu_cd <> '';

kor_poi는 시군구코드를 시도코드(2자리)와 뒤 3자리로 나눠 저장합니다. 단순히 lpad(ldong_signgu_cd,5,'0')로 5자리를 채우면 틀립니다 — 반드시 ldong_regn_cd를 앞에 붙여야 합니다.

③ 연관관광지 ↔ kor_poi 상세정보 (이름 매칭)

신뢰도: 중간 · 72~82%(표본)
select r.t_ats_nm, r.rlte_tats_nm, p.addr1, p.mapx, p.mapy, d.overview
from tar_rlte_records r
join kor_poi p
  on p.title = r.rlte_tats_nm
 and p.service = 'kor'          -- kor/korwith 중복 방지 필수
left join kor_detail d on d.content_id = p.content_id;

두 우주 사이엔 공유 코드가 없어 이름으로만 이어집니다. p.service='kor'를 빼먹으면 동일 content_id가 kor/korwith 양쪽에 있어 행이 중복됩니다. 동명 관광지가 여러 지역에 있을 수 있으니, 가능하면 지역코드도 함께 걸어 좁히세요.

④ 지역 단위 통합 (시군구 + 월 기준)

신뢰도: 높음 · 지역코드 네이티브
select
  v.signgu_code, v.region_nm, v.base_ymd,
  sum(v.tou_num) filter (where v.tou_div_cd='2') as 외지인,
  (select count(*) from tar_rlte_records r
     where r.signgu_cd = v.signgu_code and r.base_ym = left(v.base_ymd,6)) as 연관관광지_레코드수
from datalab_visitor v
where v.level = 'locgo'
group by 1,2,3;

방문자수(일별 base_ymd)와 연관관광지(월별 base_ym)는 시간 단위가 달라 left(base_ymd,6)로 맞춰야 합니다. POI 코드가 없는 혼잡도·방문자수 데이터는 이렇게 지역+시간 단위로만 다른 데이터와 결합할 수 있습니다.

05

알려진 제약사항

조인하기 전에 반드시 알아야 할 다섯 가지입니다. 위 쿼리 예시의 주석과 함께 보세요.

①

이름 매칭은 완전하지 않습니다

연관관광지↔kor_poi는 표본 기준 72~82%, 기초지자체중심↔kor_poi는 약 18%만 이름이 정확히 일치합니다. 표기 차이(약칭, 괄호 설명, 슬래시 병기)와 동명이인이 원인입니다. 매칭 실패분을 “데이터 없음”으로 처리하지 말고 별도 표시하세요.

②

content_id는 kor/korwith 서비스별로 중복됩니다

무장애(korwith) 9,960건 전부가 일반(kor) 카탈로그에도 같은 content_id로 존재합니다. service 조건 없이 kor_poi.content_id로 조인하면 행이 2배로 뻥튀기됩니다.

③

kor_poi의 지역코드는 두 체계가 섞여 있습니다

area_code/sigungu_code는 TourAPI 자체의 구(舊) 코드(약 51%만 채워짐)이고, ldong_regn_cd+ldong_signgu_cd가 실제 법정동 코드(거의 100%)입니다. 지역 조인엔 반드시 후자를 쓰고, 조합식(위 ②번 쿼리)을 적용하세요.

④

혼잡도·방문자수는 POI 코드가 전혀 없습니다

혼잡도(tats_cnctr)는 관광지 이름만, 방문자수(datalab_visitor)는 지역 단위 통계라 애초에 개별 장소를 가리키지 않습니다. 개별 관광지와 엮으려면 이름 매칭(혼잡도) 또는 지역코드 집계(방문자수)로만 가능합니다.

⑤

시간 단위가 테이블마다 다릅니다

연관관광지·기초지자체중심은 base_ym(월별), 혼잡도·방문자수는 base_ymd(일별)입니다. 함께 쓰려면 left(base_ymd, 6)로 월 단위로 맞추세요. area_tar_div·kakao_place는 아직 0행이라 조인 대상에서 제외하세요.
